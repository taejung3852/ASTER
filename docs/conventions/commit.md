# Git Commit Message 전략

이 문서는 ASTER 프로젝트의 변경 이력을 명확히 하고 유지보수 효율을 높이기 위한 커밋 메시지 작성 표준을 정의
(개발하면서 내가 참고하기 위해 정리)

---

## 커밋 전 체크리스트
- **하나의 커밋은 하나의 논리적 변경 사항만 담고 있는가?**
- **커밋 메시지의 제목만 보고도 변경 내용을 유추할 수 있는가?**
- **파일의 변경 내용과 메시지의 의도가 일치하는가?**
- **불필요한 공백이나 테스트용 로그 출력 코드가 포함되지는 않았는가?**

---

## 1. 커밋 메시지 작성 전략
1. 제목과 본문을 한 줄 띄워 분리
2. 제목은 50자 이내로 핵심만 작성
3. 제목 끝에 마침표(.) X
4. 제목은 '무엇을 했는지'를 명시하는 평서문 형식
5. 본문은 '무엇을', '왜' 변경했는지 설명
6. 본문의 각 줄은 72자 이내로 제한
7. 한국어 메시지를 사용
8. 불릿 포인트 사용

---

## 2. 커밋 메시지 키워드 규칙
모든 커밋 메시지의 시작은 아래 소문자 키워드로 시작하며, 뒤에 콜론(`:`)과 한 칸의 공백

| 키워드 | 설명 | 예시 |
| :--- | :--- | :--- |
| **`feat`** | 새로운 기능 추가 | `feat: Critic 반성 루프 노드 구현` |
| **`fix`** | 버그 및 오류 수정 | `fix: Classifier evidence 빈 문자열 검증 누락 수정` |
| **`docs`** | 문서 수정 (README, docs 등) | `docs: prompt_spec RISEN 패턴 적용` |
| **`refactor`** | 코드 리팩토링 (기능 변화 없음) | `refactor: LLM 인스턴스를 utils.py 싱글턴으로 중앙화` |
| **`test`** | 테스트 코드 추가 및 수정 | `test: ABSAState 필드 검증 단위 테스트 추가` |
| **`chore`** | 빌드 업무, 패키지 설정 수정 | `chore: qdrant-client 의존성 버전 고정` |

---

## 3. 피하면 좋은 습관
- **의미 없는 제목**: `update`, `fix`, `수정` 등 내용을 알 수 없는 메시지 사용
- **대량 커밋**: 수십 개의 파일을 한꺼번에 커밋하여 변경 이력 추적을 어렵게 만드는 행위
- **거짓 메시지**: 실제 코드 변경 내용과 다른 메시지를 작성하는 행위
- **불필요한 본문**: 제목만으로 충분한 간단한 수정에 과도하게 긴 본문 작성
