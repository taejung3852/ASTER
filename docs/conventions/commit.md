# Git Commit 컨벤션

이 문서는 ASTER 프로젝트의 커밋 메시지 작성 표준을 정의한다.

---

## 기본 구조

```
<타입>: <제목>

<본문 - 선택사항>
```

```
feat: Critic 반성 루프 노드 구현

- REVISE 판정 시 critic_feedback을 Classifier 프롬프트에 주입
- max_revisions 초과 시 hitl_triage 경유로 분기
- revisions 카운터 Supervisor에서 초기화
```

---

## 타입

| 타입 | 설명 | 예시 |
|---|---|---|
| `feat` | 새로운 기능 추가 | `feat: aspect-opinion-sentiment 관계 그래프 구현` |
| `fix` | 버그 및 오류 수정 | `fix: vector_rag domain 필터 누락 수정` |
| `docs` | 문서 수정 (README, docs 등) | `docs: state_definition 필드 테이블 보강` |
| `refactor` | 코드 리팩토링 (기능 변화 없음) | `refactor: LLM 인스턴스를 utils.py 싱글턴으로 중앙화` |
| `test` | 테스트 코드 추가 및 수정 | `test: Critic 반성 루프 단위 테스트 추가` |
| `chore` | 빌드, 패키지, 설정 파일 변경 | `chore: qdrant-client 의존성 버전 고정` |

---

## 제목 작성 규칙

| 규칙 | 설명 |
|---|---|
| 50자 이내 | git log에서 잘리지 않도록 |
| 마침표 없음 | 제목 끝에 `.` 붙이지 않음 |
| 평서문 형식 | "무엇을 했는지"를 명확하게 |
| 한국어 사용 | |

---

## 본문 작성 규칙

제목만으로 설명이 부족할 때 작성한다. 선택사항이다.

- 제목과 빈 줄 하나로 구분
- 불릿 포인트(`-`)로 작성
- **무엇을, 왜** 변경했는지 설명
- 한 줄은 72자 이내

## 좋은 커밋 vs 나쁜 커밋

```
✅ feat: aspect-opinion-sentiment 관계 그래프 build_graph 구현
✅ fix: Classifier evidence 빈 문자열 검증 누락 수정
✅ refactor: response.content 파싱 로직 parse_json_response로 중앙화
✅ docs: state_definition 에이전트별 필드 접근 테이블 추가

❌ 수정            ← 무엇을 수정했는지 알 수 없음
❌ fix             ← 어떤 버그인지 불명확
❌ 작업중          ← 작업 상태가 아닌 변경 내용을 써야 함
❌ 여러 가지 수정   ← 여러 작업을 하나의 커밋으로 묶으면 안 됨
❌ WIP             ← 공유 브랜치에 미완성 커밋 금지
```

---

## 커밋 단위 원칙

하나의 커밋 = 하나의 논리적 변경 사항.

논리적으로 완결된 단위마다 커밋한다. 기능이 실행 안 되는 상태로 커밋하지 않는다. 작업 중간 저장은 로컬에서만 한다(WIP 커밋을 push하지 않기).

---

## 절대 하면 안 되는 것

| 금지 | 이유 |
|---|---|
| `.env`, API 키, 비밀번호 커밋 | GitHub에 올라가면 삭제해도 히스토리에 영구 보관 |
| 대용량 바이너리 파일 커밋 (모델, 영상 등) | 리포지토리 크기 폭증 |
| `__pycache__/`, `.DS_Store` 커밋 | `.gitignore`로 처음부터 제외 |
| 의미 없는 메시지 (`수정`, `fix`, `1`) | 히스토리 추적 불가 |
| 여러 기능을 한 커밋에 묶기 | 특정 변경만 되돌리기 불가 |

---

## 커밋 전 체크리스트

```
□ 하나의 커밋에 하나의 논리적 변경 사항만 담겼는가?
□ 제목만 보고 변경 내용을 유추할 수 있는가?
□ 파일 변경 내용과 메시지 의도가 일치하는가?
□ .env, API 키 등 민감 정보가 포함되지 않았는가?
□ 불필요한 print/로그가 남아있지 않은가?
□ 코드가 실행 가능한 상태인가?
```
