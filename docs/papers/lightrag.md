# LightRAG

> arXiv:2410.05779 | "LightRAG: Simple and Fast Retrieval-Augmented Generation"
> Zirui Guo et al., HKU / BUPT (2024)

---

## Abstract

기존 RAG의 문제를 두 가지로 짚는다. 청크 단위의 flat data representation이라 엔티티 간 관계를 전혀 인코딩하지 못한다는 것, 그리고 컨텍스트 인식이 부족해 단편적인 답변이 나온다는 것이다. LightRAG는 그래프 구조를 텍스트 인덱싱과 검색에 통합하고, 이중 레벨 검색(low/high)과 증분 업데이트를 제안한다. Microsoft GraphRAG를 간접적으로 겨냥하는 논문이다. 커뮤니티를 순회하는 방식은 느리고 비싸다는 것이 LightRAG의 출발점이다.

---

## Section 1: Introduction

예시 하나로 문제를 정확히 짚는다. "전기차 증가가 도시 대기질과 대중교통 인프라에 어떤 영향을 미치나?" 같은 질문이다. 기존 RAG는 전기차·대기오염·대중교통 문서를 각각 따로 검색하고 연결하지 못한다. LightRAG가 해결하려는 것은 세 가지다. 여러 청크에 걸친 엔티티 관계를 이해하는 포괄적 정보 검색, 빠르고 저비용의 효율적 검색, 그리고 새 문서 추가 시 전체 재구성 없이 병합하는 동적 업데이트다.

## Section 3: Architecture

### 3.1 그래프 기반 텍스트 인덱싱

인덱싱 파이프라인은 세 단계다.

첫 번째는 엔티티·관계 추출이다. LLM으로 텍스트 청크에서 엔티티(노드)와 관계(엣지)를 추출한다. "Cardiologists assess symptoms to identify heart issues"라는 문장이면 `Cardiologists`와 `Heart Disease`가 엔티티고, "Cardiologists diagnose Heart Disease"가 관계가 된다.

두 번째가 핵심이다. **LLM Profiling**, 즉 K-V 쌍 생성이다. 각 엔티티 노드와 관계 엣지마다 LLM이 (K, V) 쌍을 만든다. K는 검색에 쓸 짧은 엔티티명 그 자체고, V는 LLM이 그 엔티티에 대해 생성한 설명 단락이다. 그리고 저장되는 것은 K(키워드)가 아니라 V(설명문)를 벡터화한 결과다.

일반 RAG와 비교하면 차이가 명확해진다.

```
일반 RAG:   "이어폰 배터리가 너무 빨리 닳아요. 충전도 느리고..."
            → 원문 그대로 벡터화 → 저장

LightRAG:   K = "배터리"
            V = "배터리는 47개 리뷰에서 부정적으로 언급된 aspect.
                 주요 표현: '닳아요', '방전', '충전 느림'"
            → V(설명문)를 벡터화 → 저장
```

키워드 자체를 벡터화하는 게 아니라, LLM이 그 키워드에 대해 생성한 설명 단락을 벡터화한다. upsert는 insert+update를 합친 DB 용어로, "있으면 덮어쓰고 없으면 새로 넣기"다.

세 번째는 중복 제거(Deduplication)다. 다른 청크에서 중복 추출된 엔티티·관계를 병합해 그래프 크기를 최소화한다.

### 3.2 이중 레벨 검색

쿼리를 구체적인 것과 추상적인 것으로 나눠 다르게 처리한다.

| 쿼리 유형 | 예시 | 검색 레벨 |
|---|---|---|
| Specific (구체) | "누가 『오만과 편견』을 썼나?" | Low-level: 특정 노드/엣지 직접 조회 |
| Abstract (추상) | "AI가 현대 교육에 어떤 영향을 미치나?" | High-level: 여러 엔티티에 걸친 관계/테마 집계 |

Low-level "직접 매칭"이 일반 RAG와 다른 점은 이렇다. 일반 RAG는 수천 개 청크 전체를 대상으로 쿼리와 유사도를 계산해 top-k를 찾는다. LightRAG low-level은 쿼리에서 "배터리"라는 키워드를 추출하면, 미리 인덱싱된 엔티티 노드 중에서 키가 "배터리"인 노드의 설명문(V)을 바로 찾아간다. 청크 전체를 뒤지지 않고 엔티티 이름으로 직행하는 것이다.

### 3.4 비용 비교

GraphRAG가 쿼리당 610,000 토큰을 쓰는 것과 대조적으로, LightRAG는 키워드 생성에 100 토큰 미만과 API 한 번 호출이면 된다. GraphRAG는 커뮤니티를 전부 순회하는 brute-force고, LightRAG는 벡터 검색으로 엔티티를 직접 찾는 구조이기 때문이다.

## Section 4: Evaluation

여러 데이터셋에서 GraphRAG를 포함한 기존 방법들과 비교했고, 대부분의 항목에서 LightRAG가 GraphRAG를 이겼다. 특히 Diversity에서 차이가 컸다.

### Ablation — -Origin 실험

LightRAG는 원래 검색 결과로 세 가지를 LLM에 넘긴다. ①엔티티 설명문(V), ②관계 설명문(V), ③원본 텍스트 청크. `-Origin` 실험은 ③을 빼고 ①②만 쓴 버전이다. 결과가 흥미롭다. 성능이 떨어지지 않고 오히려 Agriculture/Mix에서 개선됐다.

원문에는 답과 무관한 문장들이 섞여 있어 노이즈가 된다. LLM이 핵심 정보만 추출해 만든 설명문이 더 정제되어 있기 때문에 원문이 없어도 성능이 유지되는 것이다.

ASTER에 적용하면 이렇게 된다. 현재 `vector_rag.py`는 원본 리뷰 텍스트를 그대로 저장한다. `-Origin 발견`은 원문 대신 LLM이 생성한 aspect 설명문을 저장하는 것이 더 낫다는 근거다.

```
현재:  "배터리 너무 빨리 닳고 충전도 느려요. 디자인은 마음에 드는데..." → 저장
변경:  "배터리: 47건 부정 언급. 주요 표현 '닳아요'(31건), '방전'(16건)" → 저장
```

"디자인은 마음에 드는데"같은 문장이 배터리 검색 시 노이즈로 작동한다. 설명문은 배터리 관련 내용만 담는다.

## Section 5: Related Work

GraphRAG의 두 가지 한계를 명시한다. 새 정보 추가 시 커뮤니티 전체를 재구성해야 하는 동적 업데이트 불가 문제, 그리고 모든 커뮤니티를 순회하는 비효율적 brute-force 검색이다.

---

## GraphRAG vs LightRAG 구조 비교

| | Microsoft GraphRAG | LightRAG |
|---|---|---|
| 인덱스 단위 | 커뮤니티 요약 (집계된 보고서) | 엔티티/관계 설명문 (K-V 쌍) |
| 검색 방식 | 모든 커뮤니티 순회 (map-reduce) | 벡터 DB로 키워드 직접 매칭 |
| 쿼리 강점 | 글로벌 센스메이킹 | 구체+추상 둘 다 |
| 증분 업데이트 | 커뮤니티 전체 재생성 필요 | 그래프에 병합만 |
| 검색 비용 | 610,000 토큰 | <100 토큰 |

## ASTER 관점

ASTER가 LightRAG에서 차용하는 것은 LLM Profiling, 즉 K-V 쌍 생성 아이디어 하나다. `build_graph()`에서 aspect/opinion 노드를 만들 때, 각 노드에 LLM이 생성한 설명문(V)을 붙이고 그것을 Qdrant에 저장한다.

```python
{
  "key": "배터리",
  "value": "배터리는 47개 리뷰에서 부정적으로 언급됨. "
            "주요 표현: '닳아요'(31건), '방전'(16건), '충전 느림'(12건). "
            "sentiment: NEG",
  "domain": "이어폰",
  "node_type": "AspectTerm"
}
```

이중 레벨 검색과 증분 업데이트는 쓰지 않는다. ASTER는 추론 시 쿼리가 없는 배치 처리 구조라 이중 레벨 검색이 불필요하고, 배치로 전체 재처리하는 구조라 증분 업데이트도 필요하지 않다.
